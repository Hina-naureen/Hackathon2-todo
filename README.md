# Evolution of Todo — Phase I: Console App

**Hackathon II** | Spec-Driven Development & Cloud-Native AI
**Phase I Status:** Complete — 107 tests passing

---

## Project Overview

A command-line todo application built entirely through **Spec-Driven Development (SDD)**.
Every line of code was generated by Claude Code from formal specifications — no manual
coding at any step. The architecture is designed to evolve across five phases, from a
simple in-memory console app to a cloud-native, AI-powered, Kubernetes-deployed system.

---

## Features

| # | Feature | Menu Key | Description |
|---|---------|----------|-------------|
| 1 | **Add Task** | `1` | Enter a title (required, max 200 chars) and optional description (max 500 chars). Each task gets a unique auto-incremented ID. |
| 2 | **View Tasks** | `2` | Display all tasks in a formatted table — ID, status icon, title, description. |
| 3 | **Update Task** | `3` | Edit a task's title or description by ID. Press Enter to keep the existing value. |
| 4 | **Delete Task** | `4` | Remove a task by ID with a `y/n` confirmation prompt. |
| 5 | **Toggle Complete** | `5` | Flip the status of a task between `[ ]` pending and `[x]` complete. |
| 0 | **Exit** | `0` | Quit the application cleanly. |

## Setup

### 1. Install UV

UV manages the virtual environment and runs all project commands.

```bash
# Windows (PowerShell)
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh
```

Verify the installation:

```bash
uv --version
```

### 2. Clone the repository

```bash
git clone <repo-url>
cd Hackathon2-todo
```

### 3. Install the project

```bash
uv sync
```

This creates a `.venv/`, installs `pytest` for development, and registers the `todo`
command. No third-party runtime packages are downloaded.

## Spec-Driven Development Workflow

This project follows a strict SDD lifecycle. **No code is written without a spec.**

### The lifecycle

```
Constitution → Spec → Architecture → Red → Green → Refactor
```

| Stage | What happens | Output |
|-------|-------------|--------|
| **Constitution** | Define project-wide binding rules and principles | `specs/constitution.md` |
| **Spec** | Write user stories and acceptance criteria for each feature | `specs/features/*.md` |
| **Architecture** | Design module structure, layer contracts, and interfaces | `specs/architecture.md` |
| **Red** | Write failing tests that map to each acceptance criterion | `tests/test_*.py` |
| **Green** | Generate implementation (by Claude Code) to pass the tests | `src/*.py` |
| **Refactor** | Clean up presentation and structure without changing behaviour | same files |
| **PHR** | Record every prompt session automatically | `history/prompts/` |

### The rules

- **Spec wins over code.** If the implementation and the spec disagree, fix the code.
- **Claude Code generates all implementation.** Manual coding is forbidden.
- **One concern per prompt.** Never mix "write tests" with "implement feature" in one prompt.
- **Every significant architectural decision is recorded as an ADR.**
- **Every prompt session is recorded as a PHR (Prompt History Record).**

### Document hierarchy (precedence order)

```
specs/constitution.md          ← highest — Phase I binding rules
specs/architecture.md          ← module-level contracts
specs/features/task-crud.md    ← feature acceptance criteria
```

### Layer architecture

```
main.py  →  cli.py  →  task_manager.py  →  models.py
```

Import direction is strictly one-way. No layer may import from a layer above it.

## Error Reference

| Input | System response |
|-------|----------------|
| Empty title | `Error: Title cannot be empty.` |
| Title > 200 chars | `Error: Title must be 200 characters or fewer.` |
| Description > 500 chars | `Error: Description must be 500 characters or fewer.` |
| Non-integer ID | `Error: Please enter a valid task ID.` |
| ID not found | `Error: Task #<id> not found.` |
